# 章节工作流 (Chapter Workflow)

## 概览

每完成一章写作后，必须严格按以下四步流程执行：

```
Step 1: 写作 (Write)         → 完成章节内容
Step 2: 质量检查 (QA)        → 验证质量、修复问题
Step 3: 同步 tasks.md (Sync) → 更新任务追踪
Step 4: Git 提交 (Commit)    → 版本控制
```

**原则**：四步严格顺序执行，每步完成确认后才进入下一步。

---

## Step 1: 写作 (Write)

> 此步骤由 `/write` 命令负责，详见 `.claude/commands/write.md`

### 1.1 上下文加载

写作前必须加载以下文件：

| 文件 | 用途 | 必须？ |
|------|------|--------|
| `spec/tasks.md` 对应章节段落 | 获取本章所有写作要求 | ✅ 必须 |
| `spec/creative-plan.md` 对应章节段落 | 理解整体规划（**只读**） | ✅ 必须 |
| `spec/specification.md` | 故事规格参考 | ✅ 首次会话必须 |
| `.specify/memory/constitution.md` | 创作宪法/原则 | ✅ 首次会话必须 |
| 前1-2章内容 | 确保衔接和连续性 | ✅ 必须 |

### 1.2 关键写作规则

- **信息控制**：严格遵循 tasks.md 中每章的 ✅ 揭示 / 🔍 暗示 / ❌ 保留 标记
- **伏笔埋设**：tasks.md 中标注 F-xxx 的伏笔必须自然融入情节
- **章末钩子**：每章必须有明确的悬念或钩子收尾
- **字数控制**：按 tasks.md 要求的字数范围写作

### 1.3 永久风格规则（来自历史反馈）

以下规则在 Ch8 由用户确立，此后每章必须遵循：

1. **禁止过度诗意的比喻** — 如"轻轻扎进心窝"等矫情说法
2. **情感深度要够** — 不能浅尝辄止，需要体现人物内心的真实痛苦/挣扎
3. **严格信息控制** — 不该暴露的信息绝不能通过暗示或角色反应泄露

## Step 2: 质量检查 (QA)

### 2.1 检查维度

完成写作后，按以下六个维度逐项检查：

#### A. 需求覆盖度
- 逐条对照 tasks.md 中的 **必须包含** 列表
- 检查所有 **关键情节** 要求是否都已实现
- 确认 **章末钩子** 与 tasks.md 描述一致

#### B. 伏笔验证
- tasks.md 中标注的 F-xxx 伏笔是否已埋设
- 伏笔是否自然融入，不突兀
- 与前文已埋设的伏笔是否有矛盾

#### C. 章节衔接
- 与前一章结尾的场景/时间/情感是否无缝衔接
- 跨章节的物品、台词、信息是否前后一致
- 示例：Ch13 给母女碎银 → Ch14 钱袋变轻 → Ch14 又给小偷 → 钱袋更轻

#### D. 用词与设定一致性
- **地名**：兴城（不是"X城"）
- **国名**：北魏（主角所在国）、陈汉国（黄繁的掩护身份来源国）
- **称谓**：商人家庭用"宅"不用"府"
- **人名**：核对所有出场人物名字是否正确（如"黄简""林玄影""李希音"）
- **B国问题**：tasks.md 中有时用"B国"作缩写，写作中必须替换为"陈汉国"

#### E. 风格合规
- 是否有过度诗意的比喻（参见 1.3 永久风格规则）
- 情感描写是否有足够深度
- 对话是否符合角色身份和性格
- 叙述是否简洁有力

#### F. 技术问题
- 错别字、语病检查
- 标点符号是否正确（本项目使用中文弯引号 `""` `''`）
- 分段格式是否符合 constitution.md（用 `---` 分隔场景，不用数字标记）

### 2.2 评分标准

```
10分：完美，超出预期
9分：优秀，小瑕疵
8分：良好，需优化
7分：及格，问题较多
6分以下：需要重写
```

### 2.3 修复流程

发现问题时：
1. 列出所有问题及其严重程度
2. P0 问题（逻辑矛盾、设定错误、国名错误）：**必须立即修复**
3. P1 问题（风格不佳、衔接可优化）：视严重程度决定
4. P2 问题（可优化但不影响阅读）：记录在优化记录中

### 2.4 ⚠️ 编辑文件注意事项

**关键教训（多次踩坑）**：

使用 Edit 工具修改章节文件前，**必须先重新 Read 该文件的实际内容**。

原因：系统预加载的文件内容可能与磁盘上的实际内容不一致（尤其是引号字符——文件中使用中文弯引号 `""` `''`，但预加载内容可能显示为直引号）。如果用预加载的内容作为 `old_string`，Edit 工具会因为字符串不匹配而失败。

**正确做法**：
```
1. Read 文件获取实际内容
2. 从 Read 结果中精确复制要替换的文本
3. 使用复制的文本作为 old_string
```

## Step 3: 同步 tasks.md (Sync)

### 3.1 核心原则

> **creative-plan.md 是 READ ONLY — 永远不要修改**
>
> 只更新 tasks.md。creative-plan.md 是创作蓝图，写完章节后不做任何标注。

### 3.2 更新内容

需要更新 tasks.md 中对应章节的以下内容：

#### A. 任务状态标记
```markdown
# 更新前：
- [ ] [P0] **T014** - 第14章：街上游荡验证 (4000字)

# 更新后：
- [x] [P0] **T014** - 第14章：街上游荡验证 (4000字) ✅ 已完成
```

#### B. 伏笔完成标记
如果本章有埋设伏笔（F-xxx），在对应位置加上完成标记：
```markdown
# 更新前：
    - F-013：小偷说"家里妹妹病重"

# 更新后：
    - F-013：小偷说"家里妹妹病重" ✅ 已埋设（小偷13-14岁，妹妹六岁病重，与Ch13母女呼应）
```

#### C. 完成元数据块
在任务条目末尾追加完成信息（参考已完成章节的格式）：

```markdown
  - **必须包含**：✅ 全部完成
    - [逐条列出已完成的要求]
  - **章末钩子**：[描述实际钩子] ✅
  - **完成时间**：[YYYY-MM-DD]
  - **实际字数**：约[N]字
  - **质量评分**：[X.X]/10
  - **优化记录**：
    - [修复内容1]
    - [修复内容2]
    - [亮点记录1]
    - [亮点记录2]
    - [伏笔处理记录]
  - **依赖**：T[N-1] ✅
  - **输出**：`stories/content/volume1/chapter-[N].md` ✅
```

### 3.3 同步示例（T014 完整格式）

```markdown
- [x] [P0] **T014** - 第14章：街上游荡验证 (4000字) ✅ 已完成
  - **核心任务**：[保持原文不变]
  - **关键情节**：[保持原文不变]
  - **伏笔埋设**：
    - F-013：小偷说"家里妹妹病重" ✅ 已埋设（小偷13-14岁，妹妹六岁病重，与Ch13母女呼应）
  - **必须包含**：✅ 全部完成
    - 对林玄影信息的验证
    - 陈刺史的气派（伏笔仇敌）
    - 黄简的宽容（放走小偷）
  - **章末钩子**：两人重新合作，但关系依然微妙，复仇之路正式开启 ✅
  - **完成时间**：2026-02-07
  - **实际字数**：约4000字
  - **质量评分**：9.2/10
  - **优化记录**：
    - 修正1处国名错误："B国"→"陈汉国"（与Ch5/Ch6一致）
    - [更多记录...]
  - **依赖**：T013 ✅
  - **输出**：`stories/content/volume1/chapter-014.md` ✅
```

### 3.4 编辑 tasks.md 注意事项

- tasks.md 文件很大（600+行），只 Read 需要修改的行范围
- 先 Read 已完成章节（如 T013）的格式作为模板参考
- 再 Read 待更新章节（如 T014）的当前内容
- 使用 Edit 工具精确替换目标行

## Step 4: Git 提交 (Commit)

### 4.1 标准流程

```bash
# 1. 查看状态
git status

# 2. 暂存文件（明确指定文件名，不用 git add -A）
git add stories/content/volume1/chapter-014.md
git add spec/tasks.md

# 3. 提交（使用 HEREDOC 格式确保多行消息正确）
git commit -m "$(cat <<'EOF'
完成第14章：街上游荡验证

- 黄简游荡兴城：验证林玄影信息，观察城门守卫、德成当、刺史府
- 小偷伏笔埋设（F-013）：13-14岁小偷妹妹六岁病重，与Ch13母女呼应
- 黄简讲述十年经历：老者、无用之学、"出来了"三字概括逃狱
- 末尾"我们"二字：合作让步，复仇之路正式开启
- 同步 tasks.md：T014 标记完成，质量评分 9.2/10

Co-Authored-By: Claude Opus 4.6 <noreply@anthropic.com>
EOF
)"

# 4. 验证
git status  # 确认工作区干净
```

### 4.2 提交消息规范

- **首行**：`完成第N章：[章节标题]`
- **内容**：用 `-` 列表概括本章关键内容（4-6 条）
- **末行**：`Co-Authored-By: Claude Opus 4.6 <noreply@anthropic.com>`
- **暂存文件**：只暂存本章的 `.md` 文件和 `tasks.md`，不用 `git add -A`

### 4.3 暂存范围

每次提交通常包含：
- `stories/content/volume1/chapter-[N].md` — 新章节文件
- `spec/tasks.md` — 更新后的任务追踪

不应包含：
- `spec/creative-plan.md` — 永远不修改
- `.env` 或其他敏感文件

## 常见问题与教训

### Q1: Edit 工具总是报 "old_string not found"
**原因**：文件中使用中文弯引号（`""` `''`），但你可能从预加载/记忆中复制了直引号版本。
**解决**：每次编辑前重新 `Read` 文件，从 Read 结果中复制精确文本。

### Q2: tasks.md 中用了"B国"，写作中应该怎么处理？
**规则**：tasks.md 中的"B国"是缩写，写入章节时必须替换为"陈汉国"。质量检查阶段需 grep 确认无遗漏。

### Q3: creative-plan.md 需要更新完成状态吗？
**答案**：**绝对不要**。creative-plan.md 是 READ ONLY 文档，永远不做任何修改。只更新 tasks.md。

### Q4: 章节写完发现需要修改前面已提交的章节？
**处理**：修改前章 → 新建 commit（不要 amend）→ 在 tasks.md 对应章节的优化记录中注明。

### Q5: 如何确保章节间的物品/信息连续性？
**方法**：写作前必须 Read 前 1-2 章内容。质量检查时重点关注：
- 跨章节物品（如钱袋、玉佩）
- 时间线推进
- 角色知道/不知道的信息边界
- 称谓和地名一致性

## 检查清单（快速参考）

完成每章后，逐项打勾：

```
□ 写作完成，输出 chapter-[N].md
□ 逐条检查 tasks.md 必须包含项
□ 确认伏笔（F-xxx）已埋设
□ 确认章节衔接无缝
□ 确认用词/设定一致（兴城、陈汉国、北魏、宅/府）
□ 风格规则合规（无矫情比喻、情感够深）
□ 无错别字和技术问题
□ 评分 ≥ 8.0/10
□ 问题已修复（先 Read 再 Edit）
□ tasks.md 已更新（状态、伏笔、元数据）
□ creative-plan.md 未被修改 ⚠️
□ git add 指定文件（不用 -A）
□ git commit 多行消息 + Co-Authored-By
□ git status 确认干净
```
